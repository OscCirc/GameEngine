# 整体架构概述

本项目采用典型的客户端-引擎解耦双层架构模型，以实现最大化的代码复用和明确的权责划分。

## 1. 模块划分
解决方案 (Solution) 目前包含两个子工程：

### GameEngine (引擎核心层)
- **构建输出**：`GameEngine.dll` (动态链接库)。
- **职责扮演**：提供底层系统支持、生命周期主导、窗口与渲染管线的封装。
- **宏与导出**：通过 `GE_BUILD_DLL` 宏控制 `__declspec(dllexport)`，向外暴露 `GE_API`。

### Sandbox (客户端/沙盒层)
- **构建输出**：`Sandbox.exe` (可执行文件)。
- **职责扮演**：利用引擎提供的能力（导入引擎 DLL），专注于“游戏具体应该长什么样、怎么玩”等业务逻辑。
- **依赖关系**：单向依赖，Sandbox 依赖并引用 GameEngine。

## 2. 控制反转 (Inversion of Control - IoC)
普通的 C++ 程序由开发者自己在 `main()` 函数中掌握全局。而在本引擎架构中，程序的控制权被**反转**了：

引擎剥夺了客户端写 `main()` 函数的权利，并在内部预置了主程序的生命周期流转规则。一旦程序执行，是由 GameEngine 引擎底层来驱动 Sandbox 客户端游戏代码，而非 Sandbox 决定引擎如何运行。

这一设计强制约束了所有基于本引擎开发的应用，都必须遵循引擎的统一步调。
